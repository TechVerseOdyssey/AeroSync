# v0.3.0 Refactor Plan — DDD Layering (Medium Scope)

> **Status**: Phase 1–5a 完成（rc1 已 ship at commit `b4a8e0c`），剩 5b/5c soak + tag.  
> **Target window**: ~15 engineer-days ≈ 3 calendar weeks  
> **Predecessor**: v0.2.1 (shipped + tagged)  
> **Successor**: v0.4.0 (RFC-004 WAN rendezvous + NAT traversal)

---

## 1. Why this release

`docs/ARCHITECTURE_AND_DESIGN.md` §4.4 (written before v0.2.0 shipped) prescribed splitting the codebase into `aerosync-domain` / `aerosync-core` / `aerosync-infra` crates with proper DDD aggregates. v0.2.0 shipped without that split, and the codebase is **fine** at ~5K LoC in `src/core`. So this is not a fire.

It becomes a fire **the moment RFC-004 (WAN) lands**, because that adds another ~3K lines (rendezvous client + relay client + identity + ACL + JWT) under `src/wan/` on top of the already-flat `src/core` + `src/protocols`. At that scale, module boundaries start blurring, `AppState` grows another arm, and the codebase becomes hard to onboard.

**v0.3.0 = pre-emptive cleanup so RFC-004 lands cleanly in v0.4.0.**

## 2. Scope

### In scope

- Create `aerosync-domain` crate (pure logic, no IO networking)
- Create `aerosync-infra` crate (TLS / Storage / Audit / Config / Health)
- Introduce `TransferSession` aggregate root unifying the session lifecycle currently scattered across `src/core/transfer.rs`, `src/core/server.rs`, `src/core/receipt_registry.rs`
- Introduce `Storage` trait abstraction so v0.4 can swap JSON → SQLite without touching engine code
- Fix `ResumeStore` atomic-write crash safety (real bug, hitchhike on this refactor)

### Explicitly out of scope

- ❌ WAN connectivity (RFC-004 → v0.4.0)
- ❌ Actual SQLite migration (only the `Storage` trait lands; JSON impls remain default)
- ❌ Any **public API breakage**:
  - Python SDK (`aerosync.Client`, `Receiver`, `IncomingFile`) byte-stable
  - MCP tool names + schemas byte-stable
  - CLI command surface (`aerosync send`, `aerosync receive`, etc.) byte-stable
  - Wire format byte-stable (RFC-002/003 frozen)
  - TOML config schema byte-stable
- ❌ New CLI flags, new MCP tools, new transports
- ❌ GUI (`aerosync-ui` is gone, stays gone)
- ❌ RFC-005 (signed receipts) — separate v0.4+ track if pursued

### Phase 2 — Storage trait + crash-safety fix (3 d)

Define abstractions in `aerosync-domain`:

```rust
#[async_trait]
pub trait ResumeStorage: Send + Sync + 'static {
    async fn put(&self, session_id: SessionId, snapshot: ResumeSnapshot) -> Result<()>;
    async fn get(&self, session_id: SessionId) -> Result<Option<ResumeSnapshot>>;
    async fn delete(&self, session_id: SessionId) -> Result<()>;
    async fn list(&self) -> Result<Vec<SessionId>>;
}

#[async_trait]
pub trait HistoryStorage: Send + Sync + 'static {
    async fn append(&self, entry: HistoryEntry) -> Result<()>;
    async fn query(&self, filter: HistoryFilter) -> Result<Vec<HistoryEntry>>;
}
```

In `aerosync-infra` rename existing impls:

- `ResumeStore` → `JsonResumeStore impl ResumeStorage`
- `HistoryStore` → `JsonlHistoryStore impl HistoryStorage`

In root `aerosync` crate, `TransferEngine` and `FileReceiver` switch to `Arc<dyn ResumeStorage>` and `Arc<dyn HistoryStorage>` injected via constructor (default to JSON impls for back-compat).

**Hitchhike fix**: rewrite `JsonResumeStore::put` to use atomic write (`tempfile + rename`) instead of in-place write. Add a regression test that crashes mid-write and verifies the file is either fully old or fully new state.

**Acceptance**: all existing tests pass + 1 new atomic-write crash-safety test.

## 4. Key technical decisions (lock these now)

### D1. `aerosync-domain` may depend on tokio sync primitives

**Decision**: yes, depend on `tokio::sync::{watch, mpsc}` for receipt change notifications.

**Rationale**: rejecting tokio entirely would distort `ReceiptStateMachine` into a callback-driven design that doesn't match how the rest of the codebase uses receipts. The cost of "pure no-async" domain crate is too high relative to the benefit.

**Future option**: v0.5+ could split into `aerosync-domain-pure` (no tokio) + `aerosync-domain-async` (tokio bridge), if a real embedder asks.

### D2. `aerosync-py` and `aerosync-mcp` continue depending on root `aerosync` crate

**Decision**: do **not** rewire them to depend on `aerosync-domain` directly.

**Rationale**: keeps wheel CI / maturin scripts unchanged; downstream user perception is zero. The re-export pattern in root `src/lib.rs` makes their imports work without modification.

### D3. New `SessionId`, do not retire `TaskId`

**Decision**: `SessionId` is the upper-level semantic ID (one batch transfer of N files = 1 session). `TaskId` remains the per-file data-stream ID.

**Public API**: `Receipt.session_id` is a new optional getter on Python side. `Receipt.transfer_id` (= `TaskId`) stays for backward compat.

### D4. `Storage` traits are async

**Decision**: `async_trait` over both `ResumeStorage` and `HistoryStorage`.

**Rationale**: tokio::fs is everywhere; sync trait would force `spawn_blocking` wrappers, hurting performance. The slight `async_trait` macro overhead is acceptable.

### D5. RFC-004 (WAN) work continues in parallel during v0.3.0

**Decision**: pure-doc RFC-004 refinements may continue. The `src/wan/` placeholder module **is not touched** in v0.3.0 — its final home (under `aerosync-domain`, or a new `aerosync-wan` crate, or staying in root `src/wan/`) is decided in Phase 4.
